Three Design Approaches

All target: video-game-HUD density, genre-adaptive backgrounds, personality without enterprise feel. Click to select your preference.

A

Compact Restyle (Lowest Risk)

What changes: Tighten existing Vuetify components — reduce padding/margins globally, shrink field heights, condense cards into borderless grouped sections. Keep the same component structure.

Pros

  • Smallest diff — purely CSS/SCSS changes
  • No component generator changes needed
  • Low risk of breaking existing sheets

Cons

  • Still feels like Vuetify underneath
  • Limited personality improvement
  • Card-heavy layout stays bulky even when tightened
Scope: ~2-3 files changed (SCSS only). Could ship in a day.
B

Hybrid Component Refresh (Recommended)

What changes: Replace simple field components (number, string, boolean, attribute) with lightweight custom HTML that's purpose-built for density. Keep Vuetify for datatables, autocomplete, and complex widgets. Introduce a cohesive visual language: inline labels, tighter field groups, section headers with subtle borders instead of cards.

Pros

  • Fields become ~50% more compact
  • Real personality — custom look, not "Vuetify with tweaks"
  • Backgrounds shine through without card clutter
  • Genre-adaptive: fields look good on any background

Cons

  • Touches component generators + SCSS
  • Medium scope — need to update ~8-12 base components
  • Existing published sheets will look different
Scope: Base component generators + SCSS. Most field types get lighter templates. Vuetify stays for complex widgets.
C

Full Visual Overhaul (High Ambition)

What changes: Everything in B, plus: redesign the page/tab navigation, add a collapsible sidebar for quick stats, rethink how sections flow (masonry-style or CSS grid instead of flexbox rows). New section header treatment, custom scrollbar, refined color system.

Pros

  • Truly distinctive look — competitive with hand-built Foundry sheets
  • Better space utilization with grid layout
  • Sidebar for "always visible" stats (HP, key attributes)

Cons

  • Large scope — touches layout generators, sheet app, components, SCSS
  • Risk of scope creep
  • Harder to maintain across all the different field types
Scope: Sheet application generator + all component generators + SCSS + layout system. Major effort.
